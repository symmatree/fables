# ODM Maps Data Collection Process

## Overview

This document describes the process for adding and maintaining documentation for ODM (OpenDroneMap) orthophoto/map outputs. ODM map output lives on the NAS under `\\raconteur\datasets\odm-maps\{map_name}` (e.g. `bond_ave-2026-02-16`, `bond_ave-2026-02-21-house`). Each map has a doc in the facts repo at `Datasets/odm-maps-{map_name}/odm-maps-{map_name}.md`, with an optional map preview PNG in the same directory. This process is separate from [[imagesets-data-collection]] (which covers raw drone imagesets and EXIF-derived summaries).

**When following this process:** Do not record dates (the map is static data). Preserve existing commentary; only update the sections this process owns (see Document Structure and Independence below).

---

## Quick reference: ODM run record script

To fill or refresh the "ODM run record" section in a map doc:

1. **NAS must be mounted** (e.g. `/mnt/d` for `\\raconteur\datasets`). See Prerequisites below.
2. **Run** (from a directory where the polisher repo is available, e.g. workspace root):
   ```bash
   polisher/.venv/bin/python polisher/facts/collect_odm_run_record.py /mnt/d/odm-maps/<map_name>
   ```
   Replace `<map_name>` with the exact folder name on the NAS (e.g. `bond_ave-2026-02-21-house-2`).
3. **Paste** the full script output (all markdown tables) into the map doc under a level-2 heading: `## ODM run record`. Replace any existing content under that heading; do not edit other sections.

**Required inputs:** The map directory on the NAS must contain `log.json` (at root) and `odm_report/stats.json`. These are present in the standard ODM/WebODM "all" export. The script also requires `polisher/facts/odm_option_defaults.json` (next to the script); that file is part of the polisher repo.

## Prerequisites

- **NAS mounted in WSL**: Datasets share at e.g. `/mnt/d` for `\\raconteur\datasets`. If D: is mapped in Windows but `/mnt/d` is empty in WSL, run: `sudo mount -t drvfs D: /mnt/d`. See [[imagesets-data-collection#Prerequisites]].
- **Polisher repo**: Scripts live in the **polisher** repo under `facts/`. Use the polisher repo root when invoking: `polisher/.venv/bin/python polisher/facts/<script>.py`. If facts and polisher are in different trees, use absolute paths: `{POLISHER_ROOT}/.venv/bin/python` and `{POLISHER_ROOT}/facts/<script>.py`.
- **Python (polisher venv)**: Map preview needs rasterio, matplotlib. Run-record script needs only the standard library and the `odm_option_defaults.json` file next to it.

## Data and Doc Locations

- **On NAS:** `{NAS_MOUNT}/odm-maps/{map_name}` — the ODM run output (e.g. `odm_orthophoto.tif`, report, etc.). Typical WSL path: `/mnt/d/odm-maps/bond_ave-2026-02-16`.
- **In facts repo:** `Datasets/odm-maps-{map_name}/` — one directory per map. Inside it: `odm-maps-{map_name}.md` (the doc) and optionally `map_preview.png` (or another name you embed).

## Document Structure

An ODM map is an **artifact** of a single run. Map names are arbitrary (e.g. `bond_ave-2026-02-16`, `house-try-3`) and many maps may cover the same area; the structure to infer is that **the same imagesets can be run with different ODM configs to produce different maps**. This process supports collecting information to assess which combinations of collection (imagesets) and processing (config) produce useful, reliable maps.

Each ODM map doc in `Datasets/odm-maps-{map_name}/` should include:

- **Heading**: `# odm-maps / {site} / {map_name}` (e.g. `# odm-maps / bond_ave / 2026-02-21-house`).
- **Path on NAS**: `\\raconteur\datasets\odm-maps\{map_name}` (exact path to this map's output directory).
- **Input imagesets**: Links to the imageset(s) used as input for this run (e.g. `* [[imagesets-bond_ave-2026-02-21-grid]]`). Same imagesets + different config = a different map; record what this run used.
- **Map preview**: A dedicated place for the generated preview: `![[map_preview.png]]` and a footnote linking the preview to this process, e.g. `Preview generated by [[kb/odm-maps-data-collection]].` No date.
- **ODM run record**: Optional section filled by running the run-record script (see below). Contains: non-default ODM options; processing statistics (capture window, area, OpenSfM step times); features statistics; reconstruction counts; reprojection errors; GPS errors; ODM total time and GSD. Run the script and paste the markdown output into the doc.
- **Commentary**: User-written sections such as Qualitative, Processing Notes, Dataset Summary, benchmarks, etc. **Do not edit or "fix" these.** Do not change links or wording in content written by the user.

**Independence:** This process only creates or updates the sections it owns (Path on NAS, Input imagesets, Map preview and its footnote, ODM run record, and any other dedicated data-collection subsection). Never edit commentary or other hand-written sections. Each data collection process is self-contained and updates only its own sections so they do not conflict with each other or with the user.

## Map Preview Script

Run from WSL (or wherever the NAS path is available). Use absolute paths to the polisher venv and script if facts and polisher are in different trees (see Prerequisites):

```bash
{POLISHER_ROOT}/.venv/bin/python {POLISHER_ROOT}/facts/render_odm_map_preview.py <map_output_dir> <output_png>
```

- **&lt;map_output_dir&gt;**: Path to the ODM output on the NAS (e.g. `/mnt/d/odm-maps/bond_ave-2026-02-21-house`) or a copy of it. The script looks for `odm_orthophoto.tif` (or `odm_orthophoto/odm_orthophoto.tif`) and, for a boundary overlay, a GeoJSON file (see below).
- **&lt;output_png&gt;**: Path in the facts repo, typically the same directory as the map doc (e.g. `Datasets/odm-maps-bond_ave-2026-02-21-house/map_preview.png`). Embed in the doc with `![[map_preview.png]]` and add the preview footnote (see Document Structure).

**ODM output layout (fixed):** ODM does not write GeoJSON in the map root. It writes `odm_georeferencing/odm_georeferenced_model.bounds.geojson` (model bounds, same CRS as the ortho) and `odm_report/shots.geojson` (camera positions). The preview script uses the bounds file when present to draw the boundary overlay; no need to "check" each run—the layout is the same for every ODM run. If you ever add a `boundary.geojson` or `footprint.geojson` in the map root (e.g. from a different pipeline), the script will prefer that over the ODM bounds.

Requires: rasterio, matplotlib. When running headless, set `MPLCONFIGDIR` to a writable path if needed (e.g. `export MPLCONFIGDIR=/tmp`). Index: [[data-collection]].

## ODM run record script (polisher/facts/collect_odm_run_record.py)

**What it does:** Reads the ODM output directory for a single map. From `log.json` it extracts options and compares them to `odm_option_defaults.json` (in the same directory as the script); options that differ from default are listed, and options not in the defaults file are included with a `(?)` marker so they are not silently dropped. From `odm_report/stats.json` it extracts processing statistics, features statistics, reconstruction counts, reprojection errors, GPS errors, and ODM total time/GSD. It prints markdown tables (level-3 headings) to stdout.

**Command:**

```bash
polisher/.venv/bin/python polisher/facts/collect_odm_run_record.py <map_output_dir>
```

If polisher is not a sibling of your current directory, use the full path to the polisher repo root: `{POLISHER_ROOT}/.venv/bin/python {POLISHER_ROOT}/facts/collect_odm_run_record.py <map_output_dir>`.

- **&lt;map_output_dir&gt;**: Full path to the map folder on the NAS, e.g. `/mnt/d/odm-maps/bond_ave-2026-02-21-house-2`. The folder must contain:
  - `log.json` (at root)
  - `odm_report/stats.json`
  These files are present in the standard ODM/WebODM "all" export; do not use a directory that was manually stripped or that comes from a different export format.
- **Output:** Markdown (level-3 headings and tables) to stdout. Paste the entire output into the map doc under `## ODM run record`, replacing any previous content under that heading.

**Defaults file:** The script loads `odm_option_defaults.json` from the same directory as the script (`polisher/facts/`). That file must exist; it lists ODM option names (snake_case) and their default values so the script can show only non-default options. It is generated from NodeODM's schema (see `odm_options_schema.json` in the same directory; source: `https://raw.githubusercontent.com/OpenDroneMap/NodeODM/master/tests/odm_options.json`). To regenerate the defaults file, run a conversion that maps schema keys from `--kebab-case` to `snake_case` and coerces default values to int/float/bool, then write `odm_option_defaults.json`.

## Update Process

### New maps

1. **Discover maps:** List ODM map output directories on the NAS (e.g. `ls /mnt/d/odm-maps`) and/or existing doc dirs in the facts repo (`Datasets/odm-maps-*`). Identify any new map that does not yet have a doc.
2. **For each new ODM map:** One doc per map; map names are arbitrary (e.g. bond_ave-2026-02-16, house-try-3). Do not update an existing map doc when adding another—each run is its own artifact.
   - Create `Datasets/odm-maps-{map_name}/odm-maps-{map_name}.md` with the structure above (heading, Path on NAS, Input imagesets, Map preview + footnote, and leave Commentary for the user or minimal placeholders). Do not edit existing commentary.
   - Run the map preview script (see Map Preview Script). Embed the image and add the preview footnote.
   - Run the ODM run record script (see Quick reference and ODM run record script). Paste the full output under `## ODM run record` in the map doc.
   - Add the doc to `kb/things.md` under the **ODM maps** subsection: `- [[Datasets/odm-maps-{map_name}/odm-maps-{map_name}]]`.

3. **Maintain [[things]]:** In the Datasets section of `kb/things.md`, keep ODM map links under "ODM maps". Imageset links are maintained by [[imagesets-data-collection]].

### Refreshing ODM run records (existing maps)

Use this when you want to refresh or add the "ODM run record" section for maps that already have a doc.

1. **Confirm NAS is mounted** at the path you will use (e.g. `/mnt/d`). Map folder names on the NAS must match the map name in the doc (e.g. `bond_ave-2026-02-21-house-2`). To see which maps exist: list `Datasets/odm-maps-*/` in the facts repo, or read the "ODM maps" subsection in `kb/things.md`; the map name is the part after `odm-maps-` (e.g. `bond_ave-2026-02-21-house-2`).
2. **For each map** that has both (a) a doc in `Datasets/odm-maps-{map_name}/odm-maps-{map_name}.md` and (b) a folder on the NAS at `{NAS_MOUNT}/odm-maps/{map_name}`:
   - Run: `polisher/.venv/bin/python polisher/facts/collect_odm_run_record.py {NAS_MOUNT}/odm-maps/{map_name}` (e.g. `/mnt/d/odm-maps/bond_ave-2026-02-21-house-2`).
   - If the script errors (e.g. missing `log.json` or `odm_report/stats.json`), the NAS folder may not be a full ODM "all" export; fix the source or skip that map and note it.
   - Open the map doc and find or add the heading `## ODM run record`. Replace everything under that heading (down to the next level-2 heading or end of file) with the script output. Do not change any other sections.
3. **Optional:** After updating run records, fill the summary comparison table in [[experiments-house-model#Summary comparison table (run records)]] from the values in each map doc.

## Notes

- **Path on NAS:** Use the exact map folder name (e.g. `bond_ave-2026-02-21-house`) so the doc points to the correct ODM output. When running the run-record script, the path argument must be the full path to that folder (e.g. `/mnt/d/odm-maps/bond_ave-2026-02-21-house`).
- **Run-record script location:** The script and its defaults file live in the **polisher** repo: `polisher/facts/collect_odm_run_record.py` and `polisher/facts/odm_option_defaults.json`. Do not run it from the facts repo without a path to polisher; use `polisher/.venv/bin/python polisher/facts/collect_odm_run_record.py` from a directory that contains the polisher repo (e.g. workspace root).
- **What the run record contains:** The script output includes, in order: ODM options (non-default only, plus unknown options marked with `(?)`); processing statistics (report date, capture start/end, area, OpenSfM step times); features statistics; reconstruction (points, shots, observations, track lengths); reprojection errors; GPS errors; ODM processing (total time, GSD). All of that is markdown (level-3 headings); paste it in one block under `## ODM run record`.
